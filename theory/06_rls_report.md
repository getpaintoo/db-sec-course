# Практика 6. Отчет по RLS

## Таблицы с RLS

- `app.tasks`
- `app.comments`
- `app.projects`
- `app.users`

## Идея политик

1. `app_user` видит только свои задачи, свои записи пользователя и безопасный набор комментариев.
2. `app_manager` видит задачи и проекты своего управленческого контура.
3. `app_admin` получает полный доступ через отдельные политики `USING (true)`.

## Используемые функции

- `app.get_current_user_id()` — определяет строковый идентификатор прикладного пользователя;
- `app.is_project_manager(project_id)` — проверяет, что текущий пользователь управляет проектом.

## Что проверяется

1. Видимость строк для `dev_alice`, `pm_bob`, `admin_diana`.
2. Ошибки на вставку или изменение чужих строк.
3. Влияние RLS на планы выполнения через `EXPLAIN (ANALYZE, BUFFERS)`.

## Вывод

RLS дает именно тот уровень изоляции, которого нельзя добиться одними `GRANT`/`REVOKE`: роли имеют доступ к таблице как к объекту, но видят только разрешенные строки.
